# Development Guidelines（開発ガイドライン）

> 最終更新日: 2026-06-16
> 関連ファイル: [repository-structure.md](./repository-structure.md) / [architecture.md](./architecture.md)

## 1. 本書の目的

コーディング規約・開発フロー・品質維持の運用ルールを定義する。

---

## 2. コーディング規約

- **TypeScript**: `strict: true` を基本とする。`any` の使用を避け、特にOpenAI APIレスポンスのパース部分は Zod 等での型ガード/バリデーションを行う
- **ESLint**: `eslint-config-next` をベースに、未使用変数・import順序等の基本ルールを有効化する
- **Prettier**: フォーマットを統一する（セミコロンあり・シングルクォート等、プロジェクト初期化時に1セットへ固定）
- **コンポーネント設計**: Server Components優先。クライアント側の状態が必要な箇所（カードのモーダル開閉等）のみ `"use client"` を付与する

---

## 3. コミット規約

- Conventional Commits形式を推奨する（`feat:`, `fix:`, `docs:`, `chore:` 等）
- 仕様書（`docs/` 配下）を更新した場合は `docs:` プレフィックスを付け、[CLAUDE.md](../CLAUDE.md) のスペック駆動運用ルールと整合させる

---

## 4. ローカル開発の始め方

1. Node.js（LTS版）・npm をインストール
2. リポジトリをclone
3. 依存パッケージをインストール（`npm install`）
4. `.env.local.example` をコピーして `.env.local` を作成し、各値を設定（[architecture.md](./architecture.md) §7 の環境変数一覧を参照）
5. Supabaseプロジェクトに接続し、`supabase/migrations/` のスキーマを適用
6. `npm run dev` でローカル起動

### バッチ処理のローカル検証

`scripts/run-batch-locally.ts`（[repository-structure.md](./repository-structure.md) §5）でバッチパイプラインを手動実行できる。OpenAI APIキーを使用するためコストが発生する点に留意し、検証時は対象記事数を絞る等の配慮をする。

---

## 5. 環境変数管理

- `.env.local.example` には変数名のみ記載し、値は空にする
- 実際の値は各開発者が個別に用意し、本番はVercel Environment Variablesで設定する
- シークレットはリポジトリにコミットしない（`.gitignore` に `.env.local` を含める）

---

## 6. AIプロンプト変更時の品質チェック方法

PRD §10の「コンサルレベルの洞察を30秒で読める形で提供する」という品質目標を運用で守るため、以下のプロセスを設ける。

1. プロンプト変更時は、過去バッチで実際に使われた記事サンプル（直近1週間分、各カテゴリ数件程度）に対し、変更前/変更後の出力を必ず比較する
2. 評価観点をチェックリスト化する:
   - 表面的な要約に留まっていないか
   - 心理・行動変化・消費行動・社会変化への言及があるか
   - 簡潔さ（[functional-design.md](./functional-design.md) §3 の文字数目安）を満たしているか
   - 世代タグ・カテゴリ分類が妥当か
3. レビュー結果は簡易記録し、品質劣化を後から追跡できるようにする
4. MVP規模では自動評価（LLM-as-a-judge等）は過剰投資の可能性が高いため、まずは人手によるサンプルレビューを基本とし、本数が増えた場合に自動化を検討する
5. 本番反映前に必ずローカルまたはステージングでバッチを手動実行し、出力を確認してからCronスケジュールに乗せる

---

## 7. テスト方針（MVP規模）

- **ユニットテスト**: AIに依存しない純粋関数（`lib/batch/ranking.ts` の選定ロジック、`lib/batch/fallback.ts` のフォールバック処理等）を優先的にテストする
- **AI分析部分**: 出力フォーマットのスキーマ検証（Zod等）はテスト対象とするが、生成内容の「正しさ」のユニットテストは行わない（6章の人手レビュー運用で代替）
- **E2E/結合テスト**: MVPでは必須としない。バッチパイプライン全体を通すスモークテスト（ローカル手動実行での確認）は最低限実施する
- **フロントエンドのコンポーネントテスト**: 優先度は低い。MVP期間中は見た目確認中心とし、Playwright等の導入は将来検討する

---

## 8. 障害対応・運用メモ

- バッチ失敗時の確認手順: Vercelログ確認 → `batch_runs` テーブル確認 → 必要に応じ手動再実行（`scripts/run-batch-locally.ts` または再Cron実行）
- 前日データへのフォールバックが発生した場合は、原因調査と再発防止のため事後確認を行うことを推奨する
