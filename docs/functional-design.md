# Functional Design（機能仕様書）

> 最終更新日: 2026-06-16
> 関連ファイル: [product-requirements.md](./product-requirements.md) / [architecture.md](./architecture.md) / [glossary.md](./glossary.md)

## 1. 本書の目的と範囲

本書は [product-requirements.md](./product-requirements.md)（PRD）で定義された「何を作るか（Why/What）」を、「どう動くか（機能仕様）」に落とし込むドキュメントである。

- **扱う範囲**: 画面構成、ニュース記事のデータ構造、バッチ生成パイプラインの機能フロー、カテゴリ別の選定ロジックの考え方、JKトレンドの扱い、エラー時のフォールバック仕様
- **扱わない範囲**: 技術スタックの選定理由、システム構成図、DBスキーマの物理設計、デプロイ方式 → [architecture.md](./architecture.md) を参照
- **扱わない範囲**: ディレクトリ構造・命名規則 → [repository-structure.md](./repository-structure.md) を参照

## 2. 画面構成

### 2.1 トップページ

カード形式で以下9カテゴリを一覧表示する（[glossary.md](./glossary.md) の用語定義に準拠）。

1. ビッグニュース
2. α世代
3. Z世代
4. ミレニアル世代
5. X世代
6. バブル・団塊Jr世代
7. シニア世代
8. ビジネスマン
9. JKトレンド（※プレースホルダー機能。4章参照）

各カードには、カテゴリ名・カテゴリの簡潔な説明・最終更新日時を表示する。

### 2.2 カテゴリ別記事一覧

カテゴリカードを選択すると、そのカテゴリに属する記事3件をカード一覧で表示する。各カードには `title` と `summary` を表示する。

### 2.3 記事詳細表示

記事カードを選択すると、以下を展開表示する。

- 要約（summary）
- 背景感情（background_emotion）
- 欲望（desire）
- 時代の流れ（trend_direction）
- マーケティング示唆（marketing_insight）
- 元記事へのリンク（source_url）

> **未決定事項**: 詳細表示をモーダルにするか、別ページ（`/category/[category]/[id]`）にするかは実装フェーズで決定する。MVPでは実装の簡易さを優先し、まずはモーダル方式を第一候補とする。

### 2.4 最終更新日時の表示

全画面共通で「本日 8:00 更新」のような更新日時表示を行う。バッチ処理が失敗し前日データを表示している場合は「前日のデータを表示中」等、フォールバック発生を明示する（7章参照）。

## 3. ニュース記事のデータ構造

| フィールド | 型 | 説明 |
|---|---|---|
| `id` | string | 記事の一意識別子 |
| `title` | string | タイトル |
| `summary` | string | 要約（目安: 2-3文、80-120字程度） |
| `background_emotion` | string | 背景感情（目安: 1-2文） |
| `desire` | string | 欲望（目安: 1文） |
| `trend_direction` | string | 時代の流れ（目安: 1-2文） |
| `marketing_insight` | string | マーケティング示唆（目安: 1-2文） |
| `generation_tags` | string[] | 関連する世代タグ（複数可。[glossary.md](./glossary.md) のenum値） |
| `category` | string | 表示カテゴリ（`big_news` / `businessman` / `jk_trend` / 各世代enum値） |
| `source_name` | string | 情報ソース名（例: NHK, 日本経済新聞） |
| `source_url` | string | 元記事URL |
| `published_at` | datetime | 元記事の公開日時 |
| `analyzed_at` | datetime | AI分析実施日時 |
| `batch_date` | date | このバッチ生成対象日（表示制御・履歴管理のキー） |
| `rank` | number | カテゴリ内表示順位（1-3） |
| `is_fallback` | boolean | 前日データ等の代替表示かどうか |
| `source_type` | string | データ取得方式（`rss` / `manual` / `sns`。4章参照） |

「コンサルレベルの洞察を30秒で読める」というPRD要件（[product-requirements.md](./product-requirements.md) §10）を満たすため、各テキストフィールドの文字数目安を上記の通り定める。実装時のAIプロンプト設計でこの目安を出力制約として組み込む。

## 4. バッチ生成パイプラインの機能フロー

毎朝8:00（[architecture.md](./architecture.md) のVercel Cron設定に従う）に以下を実行する。

1. **RSS収集**: 登録済みRSSフィード一覧（[repository-structure.md](./repository-structure.md) `lib/rss/sources.ts`）から記事メタデータ（タイトル・URL・公開日時・元カテゴリ）を取得
2. **重複排除・フィルタリング**: 同一ニュースの複数ソース掲載を排除、極端に古い記事を除外
3. **世代/カテゴリ分類**: 各記事に世代タグ・カテゴリを付与（1記事が複数世代タグを持つことを許容する）
4. **AI分析**: 分類済み記事に対し要約・背景感情・欲望・時代の流れ・マーケティング示唆を生成（OpenAI API）
5. **上位3件選定**: カテゴリごとに重要度スコアでランキングし上位3件を選定
6. **保存**: `batch_date` を付与してSupabaseに保存。過去データは上書きせず履歴として保持する

```
RSS収集 → 重複排除/フィルタリング → 世代/カテゴリ分類 → AI分析 → 上位3件選定（カテゴリ毎） → 保存
   │（ソース障害時は当該ソースのみスキップし継続）
   └─ 失敗時の扱いは 7章 参照
```

## 5. カテゴリ別選定ロジックの考え方

| カテゴリ | 選定基準 |
|---|---|
| ビッグニュース | 全世代共通の社会的重要度（速報性・影響範囲の広さ） |
| 世代別インサイト（6世代） | 記事内容と各世代の関心領域との関連度 |
| ビジネスマン向け | 経済・経営・市場動向との関連度 |
| JKトレンド | 4章のプレースホルダー仕様に従う |

重要度スコアは以下の要素から構成される想定とし、実装時にチューニング可能なパラメータとして扱う。

- 速報性（公開日時の新しさ）
- ソース信頼度（情報ソースの重み）
- カテゴリ/世代との関連度（AI推定）
- AI推定インパクト（社会的影響の大きさ）

> **未決定事項**: 重要度スコアの具体的な計算式・各要素の重み付けは実装時に確定する。

## 6. JKトレンドのプレースホルダー仕様

### 制約

PRDの情報源（NHK・日経・Yahoo・ITmedia・TechCrunch等）は一般ニュースメディアであり、女子高生発のトレンドを一次情報として直接捕捉することは困難である。

### MVP方針

以下のいずれか、または組み合わせをMVP実装とする（実装フェーズで確定）。

- 既存RSSソースの中からZ世代・若年層・学生関連と推測される記事を抽出し仮表示する
- 運営による手動キュレーション（定期的に3件を手動指定）

画面上には「実験的機能」「データソース拡充予定」等の注記バッジを表示し、分析精度の限界をユーザーに誤解させないようにする。

### 将来拡張への配慮

`source_type` フィールド（3章）で `rss` / `manual` / `sns` を区別できる構造にしておくことで、将来TikTok/Instagram等のSNSトレンドデータソースを追加する際にデータ構造の変更を最小化する。

## 7. エラー時・データ欠損時のフォールバック仕様

| 状況 | 挙動 |
|---|---|
| バッチ処理全体が失敗 | 前日の `batch_date` データを表示し続け、「最終更新: 前日」を明示（`is_fallback=true`） |
| 特定カテゴリの記事が3件未満 | 取得できた件数のみ表示。不足分は前日の該当カテゴリデータで補完してもよい |
| 個別記事のAI分析が失敗/タイムアウト | 当該記事をスキップし候補から除外。カテゴリ内が0件になる場合のみ前日データにフォールバック |

フォールバック発生は運用監視のためログに記録する（詳細は [architecture.md](./architecture.md) の `batch_runs` テーブル参照）。
